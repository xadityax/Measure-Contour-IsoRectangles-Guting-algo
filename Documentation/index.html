Optimal Divide-and-Conquer to Compute Measure and Contour for a Set of Iso-Rectangles by Ralf Hartmut Guting

Discussion on the Algorithm

The goal of the algorithm is to optimally compute the measure and contour for a set of iso-oriented rectangles present inside a frame. The Algorithm's paradigm is Divide and conquer. The algorithm describes how to calculate the measure completely while it only describes how to calculate the horizontal contour pieces. The vertical contour pieces are to be calculated by the user, therefore, we have a used an intelligent way to calculate them by rotating the rectangles by 90 degrees and running the algorithm on them. The algorithm itself is easy to understand while we are visualizing the coordinate axes and the rectangles but the article provides only an abstract implementation of the algorithm, focusing only on the general aspects while mentioning the implementation. Data structures to be used are provided but how the data structures are used to get the optimal time complexity is not clearly explained. Take nothing away from the algorithm though, it is almost genius to visualize the problem in such a way. The algorithm is in a broader sense similar for measure and contour but while implementing, it is a little more difficult to deal with the contour problem since a binary search tree needs to be used to answer a sub-interval query [q1,q2] inside a stripe's x_interval binary search tree representation. The main function is Stripes which is analogous to the main function for any divide and conquer algorithm. The base case is when there exists only one edge inside the sub-frame while in other cases, the problem follows divide and conquer. Finally the result from left and right frames are combined by the copy, blacken and concat functions. Multiple helper functions are to be used where the most used ones are for calculating intersection, union and subtraction of intervals. Measure problem is almost complete after stripes but for Contour, the BST of a stripe is queried for an edge's interval if the edge lies on the boundary of the stripe and the stripe does not enclose the edge's rectangle. This help to get the x_interval and y cordinate of the contour piece. Once contour pieces are formed, we save them and give the files as input to our visualization tool built in python.

Issues in Coding and Time Complexity Analysis

The author describes the algorithm precisely on a conceptual basis but leaves out many details in the implementation. Although this gives room for the reader to have a custom implementation, it causes issues to decide which data structures and functions are to be used in order to get the optimal time complexity. One major problem we faced while coding was that it was almost impossible to test out individual functions until all the functions were complete. This led to a very challenging debugging phase because we ended up with roughly 600-700 lines of code, and it all had to be debugged as a whole. Another problem was because of the multiple custom data structures being used, if any binary search algorithm or sorting algorithm had to be run on them, a new comparator had to be written to decide the ordering of elements. There are, therefore, more than 15 custom comparators in the entire code. Since we worked as a team of 4, we also had to work on functions individually while maintaining cross-talk in order to use the same data structures. Later, we had to split into two subgroups to work on measure and contour. Finally, since we started out with using vectors and wrote the entire code using vectors and custom sorting, it was almost impossible to move the time complexity lower than O(N log N *log N) since we were searching 'N' times in a vector using binary search that takes O(log N). Therefore, the helper functions for stripes took O(N log N), and stripes ran for log N iterations to move the final complexity to O(N log N*log N). For contour, we could not understand how the sub-intervals query had to be answered in constant time initially, but, in the end, we figured it out. Overall, both the algorithms run in O(N log N * log N) time, but they are slow because of multiple function calls each of O(N log N) or O(N). The constant for O(N log N * log N) for this algorithm would be pretty high, is our guess. Finally, we understand how to take the complexity down to just O(N log N), but, having spent a lot of time already on the algorithm, we decided to shift focus to visualization and pristine documentation of the algorithm.
